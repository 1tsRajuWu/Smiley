# Smiley v12.0.4

**Patch release** — fixes music still not syncing to Discord after v12.0.3 (Apple Music, Spotify, etc.).

## Root cause (why v12.0.3 wasn't enough)

v12.0.3 fixed adapter path resolution and added `music_nudge()`, but the nudge ran **before** `status.activity_id` was set to `listening` and **before** `status.connected` was synced from Discord. `music_listening_active()` therefore returned false on every Listening select — the instant probe never fired. The background thread could eventually pick up tracks (after up to 8s idle sleep), but users saw only the static "Listening to music" template.

## What's fixed

- **Listening select order** — set `activity_id` early (gaming guard), sync `connected`, then `music_nudge()` so the MediaRemote probe runs immediately
- **Adapter paths** — canonical absolute paths for perl/framework (required by mediaremote-adapter)
- **Adapter health** — use `test` entitlement check instead of non-empty `get` output (stream starts even when nothing is playing yet)
- **Paused tracks** — `nudge_now_playing` returns paused metadata from MediaRemote (not playing-only)
- **Faster wake** — music thread idle sleep 8s → 2s when Listening is inactive
- **Logging** — stream stderr / exit code and `music_apply_track` failures written to `~/Library/Application Support/Smiley/logs/smiley.log`
- **Discord dedup** — RPC signature includes `Listening` activity type so template→track updates are not skipped

## How to verify (macOS + Apple Music)

1. Install **v12.0.4**, open **Discord desktop** first.
2. Smiley → **Connect** → select **Listening to music** (Settings: **Live music** on).
3. Play a song in **Apple Music** — Discord profile should show track title + artist within ~1s.
4. Optional log: `tail -f ~/Library/Application\ Support/Smiley/logs/smiley.log` — expect `mediaremote adapter at …` and `starting MediaRemote stream`.

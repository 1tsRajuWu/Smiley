# Smiley v12.0.7

**Patch release** — custom GIF presence priority and custom activity delete.

## Fixes

### 1. Custom GIF shows "Playing Rust" instead of your activity
- **Root cause:** The background live-gaming thread could race with manual activity picks (same pattern as the v12.0.6 music bug). `is_gaming_activity("")` also treated an empty slot as gaming, and the Rust game needle used substring matching (`contains("rust")`), so dev tools like `rustc` / `rust-analyzer` could false-positive as the Rust game.
- **Fix:** `refresh_live_presence` now bails out for manual presence slots (custom GIF, food, chill, work, social, etc.). `gaming_slot_active()` re-checks immediately before probed-game Discord writes. Gaming probe only runs for explicit Gaming activities or ongoing `live-*` sessions (v7 parity). Rust detection uses exact process-name match like v7.

### 2. Cannot delete custom GIF activities
- **Root cause:** `remove_custom` went through `save_config`, whose guard restored a cleared `custom` list when deleting the last activity. The UI also kept stale `categories` after delete because only `config` was updated — the grid cache never invalidated.
- **Fix:** `remove_custom` saves directly (bypassing the empty-list guard). Delete refreshes a full snapshot, invalidates the grid cache, and the delete button stays visible without hover.

## Verification

```bash
cd Smiley.v12 && npm run build
cd Smiley.v12/src-tauri && cargo test && cargo check
```

### Custom GIF presence
1. Connect → enable **Live gaming** in Settings (default on).
2. Create a custom GIF activity and select it.
3. Discord should show your custom details/GIF — not "Playing Rust" or another probed game.
4. With **Gaming** selected, probed games should still appear as before.

### Custom delete
1. **My Activities** → hover a custom tile → click **×** (always visible at reduced opacity).
2. Confirm delete — tile disappears immediately.
3. Delete the last custom activity — list stays empty after restart.
